## Design decisions logged

**Idempotency insert order** (insert AFTER success, not before): if handler fails partway, no dedupe record exists, so Stripe's retry will re-run the handler. Risk: double-processing on handler partial-success. Accepted tradeoff per brief — not over-engineering for launch.

**Unhandled events return 200**: Stripe keeps retrying if it gets a non-2xx. For event types we don't handle (e.g. `invoice.paid`), we log and return 200 immediately. This avoids Stripe filling our logs with retried deliveries for events we don't care about.

**`customer.subscription.updated` sets `"canceled"` on `cancel_at_period_end: true`**: user retains `subscription_tier` (still "premium") but `subscription_status` flips to "canceled." Frontend gating (Block 4) must read BOTH fields — `subscription_tier == "premium" AND subscription_status == "canceled"` means "still has access, but won't renew." This is confirmed business logic from Nico.

## Known edge case — "silent success" dedupe trap

**Discovered during Block 3 diagnostics (Round 4, 2026-06-15):**

`_handle_checkout_completed` can return normally with `matched_count: 0` — i.e., the DB update silently failed (no document matched the filter). The handler doesn't raise an exception, so the webhook endpoint treats it as "successfully processed" and writes the event ID to `stripe_events`. Any subsequent resend of that same event ID hits the idempotency early-return and never reaches the update logic.

**Consequence**: a broken handler (bad filter, wrong field name, etc.) will "succeed" on first delivery, write a dedupe record, and then silently ignore all Stripe retries. The bug can only be caught by the `logger.warning("user not found")` line — which must be monitored, not just present.

**Post-launch hardening option**: guard the dedupe insert behind `matched_count > 0`, or raise an exception on `matched_count == 0` so Stripe retries the event rather than dropping it. Not a launch blocker once the filter bug is fixed.
