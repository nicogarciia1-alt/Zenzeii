## Decision logged — already-subscribed check

Returning 400 when a user tries to purchase again (rather than letting Stripe create a duplicate session). This is the sensible default but Nico should confirm it matches expectations for the Premium tier specifically — a user might want to upgrade from Premium to Founding Member, but the current check blocks that path since both are "already subscribed." If that scenario needs to be supported, the already-subscribed check in a future block will need refinement (e.g., only block if `subscription_tier == request.tier`, not any paid tier).

## Rollback concern — restated for Block 3 (do not lose)

**The session-abandoned rollback is NOT implemented here.** Scenario:
1. User starts founding-member checkout → `spots_remaining` decremented (e.g., 14)
2. User closes the tab / card fails / Stripe session expires
3. `spots_remaining` stays at 14 even though no one paid

Block 3 must handle `checkout.session.expired` webhook event (and optionally `payment_intent.payment_failed`) to increment `spots_remaining` back.

The immediate Stripe-API-failure rollback IS implemented (in the `except` block) — that only covers the case where Stripe's server rejects the session creation outright.
